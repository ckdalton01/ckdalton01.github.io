Patch Management Drift: Synchronizing SCCM and Intune for Consistent Endpoint Security

The Problem

Patch management drift in hybrid System Center Configuration Manager (SCCM) and Microsoft Intune environments is a common and increasingly critical issue. Many organizations leverage both tools – SCCM for on-premises infrastructure and Intune for cloud-managed devices. However, managing updates consistently across both platforms can be difficult, leading to discrepancies in patch levels and creating significant security vulnerabilities. This occurs because SCCM updates are often deployed based on a traditional schedule and infrastructure, while Intune operates with a cloud-based delivery model. This misalignment creates “drift,” where devices managed by both systems have different patch statuses.

Implementation/Steps

Here’s a breakdown of how to address and minimize patch management drift, divided into identification, remediation, and prevention.

1. Identifying Patch Version Discrepancies

The first step is to establish a baseline for comparison. We’ll use PowerShell to query both SCCM and Intune for update deployment status.

  • SCCM Query: This script retrieves a list of installed updates and their versions from SCCM. You’ll need the Microsoft.EnterpriseContentManagement PowerShell module installed and connected to your SCCM instance.

    # Requires the Microsoft.EnterpriseContentManagement module
Import-Module Microsoft.EnterpriseContentManagement

$SCSM = Get-SCCMConnection
$Updates = Get-SCCMUpdateStatus -SMSServer $SCSM.ServerAddress -SiteCode $SCSM.SiteCode

#Export to CSV for analysis
$Updates | Select-Object ComputerName, UpdateName, Version, InstalledDate | Export-Csv -Path "C:\SCCMUpdates.csv" -NoTypeInformation

  • Intune Query: This script retrieves a list of installed updates from Intune using the Microsoft Graph PowerShell module. You’ll need the Microsoft.Graph.Intune PowerShell module installed and connected to your Azure/Intune tenant.

    # Requires the Microsoft.Graph.Intune module
Install-Module Microsoft.Graph.Intune -Force
Connect-MgGraph -Scopes "DeviceManagementService.Read.All","DeviceManagementManagedDevices.Read.All"

$Devices = Get-MgDeviceManagementManagedDevice -All
$Updates = $Devices | Select-Object Name, DeviceName, OperatingSystemVersion, ComplianceState

# Export to CSV for Analysis
$Updates | Export-Csv -Path "C:\IntuneUpdates.csv" -NoTypeInformation

    Note: The ComplianceState in Intune indicates if the device is compliant with defined policies, but doesn’t directly provide update versions. Further querying may be needed depending on your compliance policy configuration.

  • Comparison & Reporting: Compare the output CSV files (SCCMUpdates.csv and IntuneUpdates.csv) to identify discrepancies. Tools like Excel or more advanced reporting solutions (Power BI) can be used to automate this process.

2. Handling Superseded Updates & Consistent Patching

  • SCCM Superseded Updates: Configure SCCM to automatically remove superseded updates during deployment. This prevents multiple versions of the same update from being installed, simplifying remediation. In the SCCM console, navigate to Software Library > Application Management > Deployment Packages. Right-click the deployment package and choose Properties > Updates > Superseded Updates. Select “Remove superseded updates”.
  • Intune Update Rings: Utilize Intune’s update rings (pilot, broad, or standard) to control update deployment speed. This allows for a phased rollout approach and mitigation of unforeseen issues. Consider a “pilot” ring for initial testing before wider deployments.
  • Co-Management: Leverage Co-Management to sync configuration and update policies between Intune and SCCM. Co-Management allows Intune to take over management of devices previously managed by SCCM.

3. Prevention: Strategic Configuration

  • Deployment Rings: Align your SCCM deployment rings with your Intune update rings. A slower SCCM deployment ring (e.g., monthly) can complement a faster Intune ring (e.g., weekly) to provide a balance of stability and currency.
  • Reporting & Monitoring: Implement robust reporting and monitoring to proactively identify and address drift. Regularly compare SCCM and Intune update statuses.
  • Third-Party Updates: Establish a process for managing third-party updates. Consider integrating third-party patch management solutions or automating update installation via Intune’s PowerShell scripting capabilities.

Best Practices/Pro-Tip

A common mistake is assuming that Intune automatically “catches up” with SCCM updates. It’s crucial to actively monitor and compare patch statuses regularly. Set up automated reporting to flag devices with significant patch version differences. Start with a small pilot group to test any new update strategies before widespread deployment.

Verification

  • PowerShell Validation: Re-run the PowerShell scripts above after implementing changes to verify synchronization and identify any remaining discrepancies.
  • Intune Portal: Review the “Updates” section within the Intune portal to confirm update status for managed devices.
  • SCCM Console: Verify update deployment status within the SCCM console.
  • Device Logs: Examine device logs (e.g., Windows Event Viewer) for update installation errors.
